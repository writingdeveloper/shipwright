# UI i18n (next-intl, URL prefix) ŌĆö Design

**Date:** 2026-06-20
**Status:** Approved (brainstorm)

## Goal

Internationalize the starter with **next-intl** and **URL-prefix routing**
(`localePrefix: "as-needed"`), so SEO/GEO works (per-language URLs + hreflang)
while the default-locale URLs (`/`, `/sign-in`, ŌĆ”) stay unchanged. Ship it as
composable infra (`@repo/i18n`) plus a demo (en + ko, a LocaleSwitcher, one
translated surface) the user extends.

## Why URL prefix + `as-needed`

Cookie-based locale gives a single URL ŌåÆ no per-language indexing, no hreflang.
URL-prefix gives `/ko` its own indexable URL + hreflang alternates. `as-needed`
keeps the default locale (`en`) prefix-free, so **existing URLs, internal links,
and the 40 e2e are unaffected**; only `ko` adds `/ko/...`.

## Design

### Library
`next-intl` (the Next App Router i18n standard). Add to `@repo/i18n` + `apps/web`.

### `@repo/i18n` (new package)
- `routing` ŌĆö `defineRouting({ locales: ["en","ko"], defaultLocale: "en", localePrefix: "as-needed" })`.
- `getRequestConfig` (server) ŌĆö loads `messages/<locale>.json` for the active request.
- `navigation` ŌĆö `createNavigation(routing)` re-exporting locale-aware `Link`,
  `redirect`, `usePathname`, `useRouter`, `getPathname`.
- `locales` / `defaultLocale` / `Locale` type ŌĆö re-exported for consumers.
- **graceful**: set `locales: ["en"]` and i18n is a single-language no-op (no
  switcher rendered, no `/xx` routes); adding `"ko"` is the demo of how to grow.
- Messages live in `apps/web/messages/` (app-owned content), not the package ŌĆö
  the package owns the *mechanism*, the app owns the *strings*.

### `apps/web` structure
- Move every route under `app/` into `app/[locale]/` (home, sign-in, sign-up,
  forgot/reset-password, dashboard, privacy, terms, offline, error/not-found).
- `app/[locale]/layout.tsx` ŌĆö `setRequestLocale`, validate the param against
  `routing.locales` (else `notFound()`), wrap children in
  `NextIntlClientProvider`, set `<html lang={locale}>`.
- Root files that must stay outside `[locale]` (manifest.ts, sitemap, robots,
  icons, the route handlers under `app/api/`) remain at `app/`.

### proxy.ts integration (3 responsibilities)
Compose next-intl's middleware with the existing nonce-CSP + auth rate-limit:
1. `/api/auth/*` POST ŌåÆ rate-limit (unchanged).
2. page request ŌåÆ run `createMiddleware(routing)` to resolve/redirect the locale,
   THEN apply the nonce CSP to its response (carry the nonce + CSP headers onto
   the intl middleware's `NextResponse`, rather than `NextResponse.next()`).
3. matcher widened so `/ko/...` is covered; static/_next excluded as today.

### @repo/seo (hreflang + sitemap)
- Page `metadata` gains `alternates.languages` (hreflang) + `canonical`, built
  from `routing` so adding a locale updates SEO automatically.
- `sitemap.ts` emits each public route once per locale (default unprefixed,
  others `/ko/...`) with `alternates`.

### Demo translation (en + ko)
- `messages/en.json` + `messages/ko.json` ŌĆö the home page + shared nav/switcher
  labels translated. Other pages keep English literals; the demo shows the
  `useTranslations("home")` pattern to extend.
- `LocaleSwitcher` (client) ŌĆö switches language preserving the current path via
  `usePathname` + locale-aware `router.replace`.

## Verification (detailed QA)
- **unit** (`@repo/i18n`): `routing` shape (locales/defaultLocale/as-needed);
  messages files parse + en/ko have matching key sets (catch drift).
- **e2e**: `/` is `en` (no redirect, lang="en"); `/ko` renders Korean + `lang="ko"`;
  LocaleSwitcher round-trips and preserves path; home `<head>` carries hreflang
  alternates; existing 40 e2e still green (as-needed ŌćÆ default URLs unchanged);
  axe clean on `/` and `/ko`.
- **gate**: check-types (next-intl types + [locale] params), lint, test, build
  (standalone trace + dynamic params).

## Out of scope (YAGNI)
- Translating every page (only the demo surface; pattern provided to extend).
- Locale-specific number/date/currency formatting beyond next-intl defaults.
- Per-locale content negotiation tied to `@repo/legal` jurisdiction (separate concern).
- RTL languages (no rtl locale shipped; CSS is not audited for rtl).

## Risks / notes
- **proxy 3-way composition** is the sharpest risk: the intl middleware returns
  its own redirect/rewrite response; the nonce + CSP must be set on THAT response
  (and on the request headers it forwards), or CSP breaks on `/ko`. Test both
  paths.
- Moving all routes under `[locale]/` touches every page import path; internal
  `redirect()`/`<Link>` must switch to `@repo/i18n` navigation so they stay
  locale-aware (esp. auth redirects `redirect("/sign-in")`).
- `setRequestLocale` is required in every statically-rendered route for next-intl
  + App Router; pages already render dynamically (nonce CSP), which simplifies it.
- e2e/login helpers use `/sign-in` etc. ŌĆö unchanged under `as-needed` (en).
